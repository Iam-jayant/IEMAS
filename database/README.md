# IEMAS Database Setup Guide

## Overview

This directory contains the database schema and setup instructions for the Industrial Energy Monitoring & Analytics System (IEMAS) using Supabase PostgreSQL.

## Prerequisites

- Supabase account (free tier available at https://supabase.com)
- PostgreSQL client (optional, for local testing)

## Setup Instructions

### 1. Create Supabase Project

1. Go to https://supabase.com and sign in
2. Click "New Project"
3. Fill in project details:
   - **Name**: IEMAS (or your preferred name)
   - **Database Password**: Generate a strong password
   - **Region**: Choose closest to your location
4. Wait for project initialization (~2 minutes)

### 2. Run Database Schema

1. In your Supabase project dashboard, go to **SQL Editor**
2. Click "New Query"
3. Copy the contents of `schema.sql` from this directory
4. Paste into the SQL Editor
5. Click "Run" to execute the schema

This will create:
- `meters` table - Registered energy meters
- `meter_readings` table - Time-series data from meters
- `thresholds` table - Alert threshold configurations
- `alerts` table - Generated alerts
- Indexes for efficient querying
- Row Level Security (RLS) policies
- Sample data for testing

### 3. Configure Authentication

1. In Supabase dashboard, go to **Authentication** → **Providers**
2. Enable **Email** provider (enabled by default)
3. For MVP, create a test user:
   - Go to **Authentication** → **Users**
   - Click "Add User"
   - Email: `test@iemas.com`
   - Password: Generate or set a secure password
   - Confirm the user (check "Confirmed" checkbox)

### 4. Get API Credentials

1. Go to **Project Settings** → **API**
2. Copy the following values:

```
Project URL: https://xxxxx.supabase.co
anon public key: eyJxxx...
service_role key: eyJxxx... (keep this secret!)
```

3. Update your backend `.env` file:

```env
SUPABASE_URL=https://xxxxx.supabase.co
SUPABASE_KEY=eyJxxx... (anon key)
SUPABASE_SERVICE_ROLE_KEY=eyJxxx... (service role key)
DATABASE_URL=postgresql://postgres:[PASSWORD]@db.xxxxx.supabase.co:5432/postgres
```

4. Update your frontend `.env.local` file:

```env
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJxxx... (anon key)
```

## Database Schema

### Tables

#### `meters`
Stores registered energy meters in the system.

| Column | Type | Description |
|--------|------|-------------|
| meter_id | VARCHAR(50) | Primary key, unique meter identifier |
| name | VARCHAR(255) | Display name |
| location | VARCHAR(255) | Physical location |
| modbus_config | JSONB | Modbus configuration (type, baudrate, registers) |
| created_at | TIMESTAMP | Record creation time |
| updated_at | TIMESTAMP | Last update time |

#### `meter_readings`
Time-series data collected from ESP32 devices.

| Column | Type | Description |
|--------|------|-------------|
| id | BIGSERIAL | Primary key |
| meter_id | VARCHAR(50) | Foreign key to meters |
| timestamp | TIMESTAMP | Reading timestamp from meter |
| voltage | REAL | Voltage (0-1000V) |
| current | REAL | Current (≥0 A) |
| active_power | REAL | Active power (≥0 W) |
| reactive_power | REAL | Reactive power (VAR) |
| apparent_power | REAL | Apparent power (≥0 VA) |
| power_factor | REAL | Power factor (-1 to 1) |
| frequency | REAL | Frequency (0-100 Hz) |
| cumulative_energy | REAL | Energy (≥0 kWh) |
| firmware_version | VARCHAR(50) | ESP32 firmware version |
| uptime_seconds | INTEGER | ESP32 uptime |
| created_at | TIMESTAMP | Database insertion time |

**Indexes:**
- `idx_meter_readings_meter_id` - Fast meter filtering
- `idx_meter_readings_timestamp` - Fast time-range queries
- `idx_meter_readings_meter_time` - Composite index for meter + time queries

#### `thresholds`
Alert threshold configurations per meter.

| Column | Type | Description |
|--------|------|-------------|
| meter_id | VARCHAR(50) | Primary key, foreign key to meters |
| high_power_threshold | REAL | Power alert threshold (W) |
| low_power_factor_threshold | REAL | Power factor alert threshold |
| updated_at | TIMESTAMP | Last update time |

#### `alerts`
Threshold-based alerts generated by the system.

| Column | Type | Description |
|--------|------|-------------|
| id | BIGSERIAL | Primary key |
| meter_id | VARCHAR(50) | Foreign key to meters |
| alert_type | VARCHAR(50) | HIGH_POWER or LOW_POWER_FACTOR |
| measured_value | REAL | Actual measured value |
| threshold_value | REAL | Threshold that was exceeded |
| timestamp | TIMESTAMP | When alert was triggered |
| acknowledged | BOOLEAN | User acknowledgment status |
| acknowledged_at | TIMESTAMP | Acknowledgment time |
| acknowledged_by | UUID | User who acknowledged |
| dismissed | BOOLEAN | Dismissal status |
| dismissed_at | TIMESTAMP | Dismissal time |
| dismissed_by | UUID | User who dismissed |
| created_at | TIMESTAMP | Alert creation time |

**Indexes:**
- `idx_alerts_meter_id` - Fast meter filtering
- `idx_alerts_timestamp` - Fast time-range queries
- `idx_alerts_active` - Fast active alert queries (not dismissed)

### Views

#### `latest_meter_readings`
Shows the latest reading for each meter with online/offline status.

#### `active_alerts`
Shows all non-dismissed alerts with meter information.

## Security

### Row Level Security (RLS)

RLS is enabled on all tables with the following policies:

- **Authenticated users**: Full read access to all tables
- **Authenticated users**: Full write access to meters, thresholds, alerts
- **Service role**: Can insert meter_readings and alerts (for backend API)

For production, these policies should be refined based on:
- Organization-level isolation
- Role-based access control (admin, operator, viewer)
- Time-based data retention policies

## Querying Examples

### Get latest reading for each meter
```sql
SELECT * FROM latest_meter_readings;
```

### Get readings for a specific meter in the last 24 hours
```sql
SELECT * FROM meter_readings
WHERE meter_id = 'MTR-001'
AND timestamp > NOW() - INTERVAL '24 hours'
ORDER BY timestamp DESC;
```

### Get active alerts
```sql
SELECT * FROM active_alerts;
```

### Get daily energy consumption
```sql
SELECT 
    meter_id,
    DATE(timestamp) as date,
    MAX(cumulative_energy) - MIN(cumulative_energy) as daily_energy
FROM meter_readings
WHERE meter_id = 'MTR-001'
GROUP BY meter_id, DATE(timestamp)
ORDER BY date DESC;
```

## Maintenance

### Backup

Supabase provides automatic daily backups. To manually backup:
1. Go to **Database** → **Backups** in Supabase dashboard
2. Click "Create Backup"

### Performance Optimization

For production deployments with high data volumes:

1. **Enable TimescaleDB extension** (for time-series optimization):
```sql
CREATE EXTENSION IF NOT EXISTS timescaledb;
SELECT create_hypertable('meter_readings', 'timestamp');
```

2. **Set up data retention policies**:
```sql
-- Keep only last 90 days of data
SELECT add_retention_policy('meter_readings', INTERVAL '90 days');
```

3. **Monitor query performance**:
```sql
SELECT * FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

## Troubleshooting

### Connection Issues

If backend can't connect to database:
1. Check `DATABASE_URL` in `.env`
2. Verify IP allowlist in Supabase dashboard (Database → Settings → Connection Pooling)
3. Check firewall rules

### RLS Policy Errors

If getting permission denied errors:
1. Verify user is authenticated
2. Check RLS policies in Supabase dashboard (Database → Policies)
3. Ensure service role key is used for backend API operations

### Performance Issues

If queries are slow:
1. Check index usage: `EXPLAIN ANALYZE <query>`
2. Review connection pool settings
3. Consider upgrading Supabase plan for more resources
